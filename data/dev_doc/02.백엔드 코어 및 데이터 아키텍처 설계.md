# APC 밸브 모니터링 및 제어 앱 - 백엔드 코어 및 데이터 아키텍처 (Phase 2)

---

## 1. 클래스 설계 원칙 (4사분면 매트릭스)

* 자원 점유 여부와 독립적 문맥(스레드) 소유 여부를 기준으로 클래스의 역할과 생명주기를 완벽히 분리한다.
* 이를 통해 최대 115200bps, 10~20ms 주기의 고속 통신 환경에서 스레드 안전성을 확보한다.
* **xxxxxManager (물리적 자원 O, 독립적 문맥 X)**: 앱 전체에 단 하나만 존재하는 싱글톤 객체이다.
  * 자신만의 루프 없이 내부 Lock(Mutex)을 통해 스레드 안전성을 보장하는 수동적 관리자이다.
  * 매니저 클래스의 예시로는 SerialPortManager (COM 포트 관리), ProtocolManager, TagManager, GlobalValueManager가 있다.
* **xxxxxService (물리적 자원 O, 독립적 문맥 O)**: 앱 내에 존재하는 싱글톤 객체이다.
  * 백그라운드에서 전체 시스템의 생명주기를 관리하는 엔진의 지휘자 역할을 수행한다. 예시로는 FirmwareUpdateService가 있다.
* **xxxxxHelper (물리적 자원 X, 독립적 문맥 X)**: 상태를 가지지 않는 순수 정적(Static) 메서드 모음이다.
  * 데이터 파싱 및 조립을 전담하며, 대표적인 예로 PacketHelper가 있다.
* **xxxxxWorker (물리적 자원 X, 독립적 문맥 O)**: 특정 화면이나 작업에 종속되어 필요할 때 생성되고 소멸하는 독립 스레드이자 실행 주체이다.
  * 워커 클래스의 예시로는 TagWorker가 있다.

## 2. 데이터 주도 아키텍처 (Schema & Bootstrapping)

* 800여 개의 불규칙한 전용 프로토콜과 Tag 정보를 하드코딩하지 않고, 외부 JSON 스키마 기반으로 동적 관리한다.
* **스키마 분리**: 통신 규칙을 정의하는 `protocol_schema.json`과 데이터 메타정보를 정의하는 `tag_schema.json`을 분리하여 관리한다.
* **Master-Slave 및 방어적 코딩**: PC의 요청(req)에 밸브가 응답(resp)하는 구조로 설계되어 있다.
  * 무결성 검증(체크섬)이 없으므로, 수신 시 기대 헤더가 다르면 즉시 패킷을 버린다(Drop).
  * Deadlock 방지를 위해 짧은 타임아웃(5~10ms)을 필수적으로 적용한다.
  * 특별히 예외적인 경우로 사용자가 Trace 모드를 활성화 하면 PC는 요청할수 없고 오직 밸브에서 패킷을 보내는 내용을 계속 수신만 하는 모드가 있으니 주의 해야한다.
* **부트스트래핑(Bootstrapping) 및 Hot-Swap 흐름**: 부트스트래핑 과정은 다음과 같이 진행된다.
  1. HomeWindow에서 밸브와 첫 연결시(또는 재 연결시) 펌웨어 버전 및 설정 모드에 해당하는 Tag 값을 GlobalValueManager를 통해 획득한다.
  2. 획득한 Tag 정보를 바탕으로 TagWorker를 통해 읽어오도록 한다. Tag값을 밸브로 부터 읽어와 Tag값을 설정한다.
  3. GlobalValueManager에서는 펌웨어 버전 및 설정 모드에 해당하는 Tag 값들의 변경을 감지한다.
  4. 펌웨어 버전 및 설정 모드에 해당하는 Tag 값들의 조건 조합을 분석하여 해당 밸브에 맞는 진짜 스펙의 폴더(`schemas/v3.0_advanced/` 등)를 결정한다.
  5. 결정된 진짜 스키마로 매니저들의 데이터를 갈아끼우고(Hot-Swap), 실제 모니터링용 TagWorker들을 가동한다.(펌웨어 버전 및 설정 모드에 해당하는 Tag는 갈아끼우는 데이터에 속하지 않는다.)

## 3. Tag 모델링 및 메모리 다이렉트 링킹

* 초고속 통신 병목을 없애기 위해 '문자열 검색'을 배제하고 '포인터 직접 참조' 방식을 사용한다.
* **포인터 링킹(Binding)과 PacketHelper**: 초기화 완료 시, ProtocolManager의 PacketParamModel 객체들이 대상 Tag 객체의 메모리 주소(Reference)를 직접 들고 있게 된다.
  * PacketHelper는 통신 수신 시 이 모델을 보고 문자열을 쪼갠 뒤 검색 과정 없이 포인터에 값을 다이렉트로 주입한다.
* **다양한 매핑 구조 소화**: 여러 형태의 매핑 구조를 다음과 같이 처리한다.
  * **1:N 매핑 (Bitmap)**: PacketHelper가 비트 마스킹 연산을 수행하여 여러 Tag 포인터에 각각 True/False를 주입한다.
  * **N:1 매핑 (패킷 분할)**: Shadow Tag(숨김 태그)로 파편화된 패킷 데이터를 각각 받는다.
    * 파편화된 데이터를 구독(Subscribe)하는 Computed Tag가 값이 모두 모였을 때 조합 연산을 수행하여 최종 UI 이벤트를 방출한다.
* **Tag 팩토리와 다형성**: `TagA * (TagB - 10)`처럼 예외적으로 복잡한 연산이 필요한 Tag는 BaseTag를 상속받은 커스텀 클래스(CustomPressureTag 등)로 분리한다.
  * 분리된 커스텀 클래스는 내부 Converter 로직을 통해 캡슐화를 유지한다.

## 4. TagWorker의 동기화 및 상태 머신(FSM) 설계

* 화면(UI)에 종속되지 않고, 밸브 통신 프로토콜 특성에 맞춰 최적화된 통신 스케줄을 운영한다.
* **Smart Polling**: TagWorker는 화면에서 요구하는 Tag 목록을 역추적하여, 중복을 제거한 최소한의 PacketModel 단위로 통신 스케줄(명령어 리스트)을 병합한다.
* **Lock 기반의 자연스러운 대기열(FIFO)**: SerialPortManager의 Mutex Lock을 짧게 잡고 푸는 방식으로 우선순위 큐 없이 즉각적인 Write 명령 삽입을 보장한다.
  * 기아(Starvation) 현상을 막기 위해 Read 루프 사이에 미세한 휴식(sleep)을 부여한다.
* **TagWorker 내부 상태 천이 (FSM) 및 UI 연동**: 상태 천이는 다음과 같이 이루어진다.
  * **STATE_INIT_READ**: 초기 화면 구성을 위한 전체 데이터 1회성 Read 작업을 수행한다.
  * **STATE_MONITORING**: 일반적인 고속 Polling (Read) 작업을 진행한다.
  * **STATE_WRITE**: UI의 제어 명령 발생 시 진입하며, 송신 후 Lock을 즉시 해제하여 다른 Worker의 단순 Read를 허용한다.
  * **STATE_VERIFY_READ**: 제어 직후, 변경된 값이 밸브에 적용되었는지 확인하기 위한 Read 작업을 수행한다.
* **UI 상태 연동**: Write 명령 발생 시 해당 입력창을 즉시 비활성화하고 "Writing..." 상태로 변경한다.
  * STATE_VERIFY_READ가 성공하여 값이 업데이트되면 비로소 UI 잠금을 해제한다.
